## 1. 游戏概述

《今天也要送到》是一款面向老年玩家的 3D 网页陪伴型送报游戏。玩家每天进入游戏后，先选择今天的送报角色，接受陪伴助手的问候，回答当天状态，然后选择骑自行车或步行，在温暖的日本住宅街区中把报纸送到邻居家。

游戏的核心不是速度、分数或失败惩罚，而是：

- 让老人感到有人陪伴、有人关心、有人等待；
- 通过找路、记忆住户、选择报纸、观察地标等方式自然用脑；
- 提供短时间、低压力、可重复的每日任务；
- 预留家人留言、异步陪伴、合作和友好对战扩展。

本游戏不作为医疗产品宣传，不承诺预防或治疗阿兹海默症。设计目标是支持规律参与、社交连接和适度认知活动。

## 2. 目标用户

### 2.1 主要用户

- 60 岁以上日本或华语老人；
- 喜欢慢节奏、怀旧生活题材的人；
- 操作反应较慢，但愿意轻松游戏的人；
- 独居、半独居，或家人希望通过游戏保持陪伴的人。

### 2.2 次要用户

- 老人的家人；
- 护理员、养老机构工作人员；
- 希望设计老人友好游戏的开发者。

### 2.3 用户体验目标

玩家完成一局后应感到：

- 今天有人记得我；
- 我完成了一件有意义的事；
- 我认识这个社区的人；
- 我可以慢慢来，不会被责备；
- 即使一个人玩，也有陪伴感。

## 3. 游戏定位

### 3.1 类型

- 3D 网页游戏；
- 慢节奏送达模拟；
- 轻度认知活动；
- 陪伴型日常游戏。

### 3.2 参考方向

玩法参考 FC《送报童》的“沿路送报”结构，但本项目不强调反应速度和躲避危险，而是改为老人友好的陪伴体验。

画面方向不是纯 8-bit，而是：

> 昭和末期～平成初期日本街区氛围 + 现代大字 UI + 柔和低多边形 3D。

## 4. 核心玩法循环

### 4.1 单局流程

1. 标题画面生成当日随机城市；
2. 玩家选择送报角色；
3. 点击开始，获得声音播放权限；
4. 输入或使用随机玩家名字；
5. 回答“今天状态如何”；
6. 选择骑自行车或步行；
7. 进入街区，从第一个目标附近出发；
8. 沿导航箭头前往目标；
9. 进入隐形投递范围后，头顶出现报纸图标；
10. 按空格或按钮投递报纸；
11. 报纸飞向目标，住户出现感谢反馈；
12. 继续下一户；
13. 达成 6/6 等目标后进入总结；
14. 获得今日感谢卡，可复制分享给家人。

### 4.2 长期循环

- 每天打开游戏都有不同地图、不同路线、不同角色和不同留言；
- 住户、路线、家人留言和社区细节形成熟悉感；
- 通过每日完成送报建立规律参与感。

## 5. 操作设计

### 5.1 电脑端

- `W / ↑`：前进；
- `S / ↓`：后退；
- `A / ←`：向左转；
- `D / →`：向右转；
- `空格`：投递报纸；
- 点击按钮：投递、暂停、总结、找目标、自动导航、高对比度。

### 5.2 手机端

- 左侧虚拟方向控制：左右转向；
- 右侧前进按钮：按住才前进；
- 右侧后退按钮：较小；
- 独立投递按钮：进入投递范围后会有提示动画；
- HUD 与虚拟按钮错开，避免遮挡。

### 5.3 辅助操作

- 自动导航：角色沿道路自动前进和转弯；
- 找目标：角色朝向目标方向；
- 安心模式：减速、放大提示、降低操作压力；
- 高对比度：加强目标、UI 和可读性。

## 6. 投递机制

### 6.1 可见光圈与隐形范围

每个目标房屋有两个范围：

1. 可见小光圈：显示目标投递点，颜色与房屋/任务一致；
2. 隐形大光圈：实际可投递范围，半径大于可见光圈。

当玩家进入隐形大光圈后：

- 角色头顶出现报纸图标；
- 投递按钮开始温和动画；
- 按空格或按钮后，角色自动面向目标并投出报纸。

如果未进入范围就投递：

- 不惩罚；
- 不扣分；
- 用漫画气泡温柔提示“再靠近一点点”。

### 6.2 投递表现

投递时：

- 角色做出抛报动作；
- 报纸沿弧线飞向目标；
- 报纸飞行时旋转；
- 成功后房屋轻微反应；
- 收报人头像出现，说出感谢语。

感谢语必须来自被送报的住户，而不是助手代说。

## 7. 导航系统

### 7.1 路面箭头

- 地面显示约 10 个导航箭头；
- 箭头沿道路分布；
- 箭头贴地显示，像道路标线；
- 人物、自行车、建筑可以自然遮挡箭头；
- 接近终点后箭头直接指向小光圈；
- 箭头不使用斜向穿墙路径。

### 7.2 温情导航语音/对白

导航不只用机械箭头，还用陪伴式提示：

- “这条路慢慢走就好。”
- “前方差不多要向左转。”
- “快到了，看到小光圈就可以投递。”

左右提示必须以玩家当前朝向为准，不能反向。

### 7.3 小地图

右上角显示小地图：

- 当前道路网；
- 玩家位置；
- 当前目标；
- 未完成投递点；
- 当前所在区域名。

电脑端对白避开小地图；手机端小地图自动缩小。

## 8. 场景与地图设计

### 8.1 地图参考

地图参考大阪北江口 / 井高野附近格局：

- 河川 / 河堤 / 河川敷；
- 低层住宅密集区；
- 学校与绿地；
- 日常商店；
- 公寓和团地感建筑；
- 不等距但清晰的道路网。

项目使用 OpenStreetMap / Overpass API 作为开放数据参考，不直接使用 Google Maps 受限数据。

### 8.2 河川设计

河流不是蓝色平面，而要包含：

- 低于地面的水面；
- 混凝土护岸；
- 河川敷草地；
- 河堤天端路；
- 护栏；
- 桥；
- 河边树；
- 鸭子等生物。

### 8.3 建筑类型

街区中至少包含：

- 一户建住宅；
- 木造老屋；
- 现代住宅；
- 便利店；
- 超市；
- 医院；
- 诊所；
- 药局；
- 邮局；
- 公寓；
- 大楼；
- 银行；
- 交番；
- 学校；
- 图书馆；
- 咖啡店；
- 食堂；
- 面包店；
- 理发店；
- 花店；
- 书店；
- 鱼店；
- 澡堂；
- 停车场；
- 神社与鸟居。

### 8.4 昭和末期～平成初期住宅细节

住宅应表现：

- 深色瓦屋顶；
- 雨樋；
- 铝合金窗框；
- 雨户；
- 木质外墙带；
- 电表箱；
- 门牌；
- 旧式邮筒；
- 生活阳台；
- 花盆；
- 玄关台阶；
- 小门和围墙。

### 8.5 性能与 LOD

- 近处建筑显示高细节；
- 远处建筑只保留主体和屋顶；
- 树木、窗户、屋顶、行人小部件使用 InstancedMesh 批量渲染；
- 保持建筑数量，但控制远景开销；
- 手机端优先保证稳定帧率。

## 9. 角色设计

### 9.1 玩家角色

开始界面提供至少 12 位送报角色：

- 男性 6 位；
- 女性 6 位；
- 外观包含帽子、发型、眼镜、胡子、服装、体型差异；
- 选择角色后，默认名字和对白根据性别调整。

角色必须：

- 正常朝前行走，不能横着走；
- 步行动画频率与速度匹配；
- 骑车时有车轮、脚踏、腿部动作；
- 停车时一只脚落地；
- 转向时车把和车身轻微侧倾；
- 手上或车上能看到报纸。

### 9.2 住户角色

正式原型不少于 30 位送报对象，每位包含：

- 独立名字；
- 礼貌称呼；
- 性别或夫妇类型；
- 报纸偏好；
- 房屋线索；
- 感谢语；
- 漫画头像特征。

### 9.3 陪伴助手

助手随机名字，并随语言本地化。

助手负责：

- 问候；
- 导航；
- 鼓励；
- 安全提醒；
- 轻度回忆提示；
- 解释操作。

助手不负责代替住户说“谢谢”。

## 10. UI / UX 设计

### 10.1 基础原则

- 字大；
- 按钮大；
- 信息少；
- 不闪烁；
- 不出现强失败语言；
- 保持半透明，减少遮挡；
- 重要信息优先。

### 10.2 任务卡

任务卡显示：

- 当前完成数 / 总数，例如 0/6；
- 当前目标住户；
- 报纸类型；
- 投递按钮；
- 暂停 / 总结 / 自动导航等辅助按钮。

手机端任务卡更小，只保留必要信息。

### 10.3 漫画对白

对白采用“头像 + 气泡”形式。

电脑端：

- 半透明玻璃感；
- 偏左显示，避开小地图；
- 不遮挡关键 UI；
- 文字保持高可读性。

手机端：

- 更小；
- 半透明；
- 尽量不遮挡画面和按钮。

---

## 11. 声音设计

### 11.1 音乐

- 背景音乐约 1 分钟；
- 可无缝循环；
- 轻快、温暖、有散步感；
- 避免沉闷、刺耳或高压。

### 11.2 音效

需要包含：

- 骑自行车声；
- 步行声；
- 投报声；
- 报纸落地声；
- 送达提示；
- 鸟鸣；
- 风声；
- 河边环境声；
- 行人和动物的低频声音。

音乐和声效可分别开关。

---

## 12. 多语言设计

支持：

- 日文；
- 英文；
- 繁体中文；
- 简体中文。

要求：

- 自动识别浏览器语言；
- 右上角可随时切换；
- 切换语言不刷新页面；
- 玩家名字、助手名字、住户称呼随语言变化；
- 不同语言使用适合字体；
- 中文模式下不混入日文系统提示。

## 13. 老人友好设计

### 13.1 操作友好

- 支持慢速操作；
- 投递范围宽松；
- 自动导航；
- 找目标；
- 高对比度；
- 手机大按钮。

### 13.2 心理友好

避免使用：

- 失败；
- 错误；
- 扣分；
- 你记错了；
- 赶快。

改用：

- “没关系，我们再看看。”
- “不着急，慢慢来。”
- “前面有点挤，我们先等一下。”

### 13.3 认知活动

认知内容自然融入游戏：

- 记住住户；
- 观察房屋线索；
- 分辨报纸类型；
- 规划路线；
- 回忆刚才送给谁；
- 识别地标。

不把这些包装成考试。

## 14. 交通与生活感

街道中应有：

- 行人；
- 骑车路人；
- 遛狗的人；
- 猫、狗、麻雀、鸭子、乌鸦、蝴蝶、蜻蜓等；
- 乌鸦翻垃圾；
- 河边鸭子；
- 路边树叶、花瓣。

行人和车辆规则：

- 沿道路 / 人行道移动；
- 不穿房子；
- 有碰撞判定；
- 遇到玩家会避让或停下；
- 自动导航遇到障碍时停住，不绕圈、不回头乱转。

---

## 15. 自动导航设计

自动导航不是“自动前进”，而是：

- 使用与地面箭头相同的道路 waypoint；
- 沿路自动前进；
- 到路口自动转弯；
- 接近目标后朝向投递点；
- 遇到行人或障碍先停下；
- 不反复重新规划导致原地打转；
- 投递后等待住户反馈结束，再转向下一目标。

---

## 16. 联机与家人陪伴预留

### 16.1 家人留言

家人可提前留下：

- 鼓励语；
- 今日提醒；
- 晚上打电话约定；
- 简短问候。

内置大量温情留言，避免重复。

### 16.2 异步陪伴

老人完成后，家人可以看到：

- 今日送达户数；
- 收到感谢数；
- 今日状态选择；
- 感谢卡；
- 老人留给家人的一句话。

### 16.3 合作模式

未来支持：

- 老人与家人一起送；
- 两人分头送；
- 一人导航一人投递；
- 最后在公园集合。

### 16.4 友好对战

不使用排名羞辱，只比较温和表现：

- 找房子更快；
- 投递更稳；
- 收到更多感谢；
- 两人共同完成社区目标。

## 17. 技术架构

### 17.1 当前技术

- HTML / CSS / JavaScript；
- Three.js WebGL 渲染；
- Vite 本地开发；
- GitHub Pages 静态部署；
- LocalStorage 本地保存；
- Web Audio API 音乐和音效；
- Playwright 本地自动化测试。

### 17.2 主要模块

```text
src/
├─ main.js                 # 启动入口
├─ app.js                  # 游戏流程控制
├─ i18n.js                 # 多语言
├─ buildInfo.js            # 版本号
├─ audio/                  # 音乐与音效
├─ data/                   # 住户、地图、角色、留言数据
├─ game/                   # 移动、投递、导航、难度
├─ input/                  # 键盘和触屏输入
├─ render/                 # Three.js 场景渲染
├─ state/                  # 游戏状态与本地保存
└─ ui/                     # HUD 和界面
```

### 17.3 性能策略

- InstancedMesh 批量渲染树、花草、窗户等重复物件；
- 建筑 LOD；
- 控制实时阴影；
- 远处低细节；
- 手机端降低渲染压力；
- 避免每帧重建大型数据。

## 20. 设计原则总结

1. 陪伴优先于成绩；
2. 清楚优先于复杂；
3. 慢节奏优先于刺激；
4. 温情提示优先于失败惩罚；
5. 生活感优先于竞技感；
6. 真实街区参考，但必须游戏化整理；
7. 老人能看懂、能操作、能安心完成。
