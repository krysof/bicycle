当前版本：v20260611.3  
项目类型：静态 Web / Three.js 3D 网页游戏  
部署方式：GitHub Pages  
本文目的：说明本项目的代码组织、模块职责、数据流、渲染逻辑、状态管理、输入系统、音频系统、性能策略、测试与后续扩展方向。

---

## 1. 技术目标

本项目的技术设计目标不是追求复杂引擎能力，而是支撑一个老人友好、可在浏览器直接访问、可持续迭代的 3D 陪伴型送报游戏。

技术目标包括：

1. **无需安装即可访问**：通过浏览器打开 GitHub Pages 即可玩；
2. **模块清晰**：不把所有逻辑堆在 `main.js`；
3. **低门槛维护**：使用原生 ES Modules，避免复杂构建链；
4. **老人设备可运行**：控制模型复杂度，使用批量渲染和 LOD；
5. **支持手机和电脑**：同一套游戏逻辑，分离输入和 UI；
6. **多语言可扩展**：日 / 英 / 繁中 / 简中统一走 i18n；
7. **玩法可持续扩展**：住户、角色、留言、地图和任务都以数据驱动为主；
8. **部署稳定**：版本号、缓存参数、GitHub Pages 可验证。

---

## 2. 总体架构

项目采用“轻量游戏循环 + 模块化功能层”的结构。

整体分层如下：

```text
浏览器页面
  ↓
main.js 初始化
  ↓
App 游戏流程控制
  ↓
状态层 state
  ↓
玩法层 game
  ↓
渲染层 render / UI 层 ui / 输入层 input / 音频层 audio
  ↓
Three.js / DOM / Web Audio / LocalStorage
```

核心原则：

- `main.js` 只负责启动；
- `app.js` 管理流程和模块协作；
- `game/` 只处理规则；
- `render/` 只处理 3D 表现；
- `ui/` 只处理界面；
- `data/` 尽量保存静态和生成数据；
- `state/` 作为运行时状态中心。

## 4. 启动流程

- WebGL Canvas：`#gameCanvas`；
- UI 根节点：`#ui`；
- 版本号角标：`#versionBadge`；
- 语言 / 音乐 / 声效按钮；
- 漫画气泡：`#comicBubble`；
- 手机触控按钮；
- 游戏 HUD；
- 小地图 Canvas；
- `src/main.js` 模块入口。

### 4.2 main.js

`main.js` 负责：

1. 读取 `BUILD_VERSION`；
2. 更新页面版本号；

5. 调用 `app.start()`。

`App` 构造时完成：

- 应用语言；
- 创建初始状态；
- 创建 ThreeRenderer；
- 在标题画面前准备随机世界；
- 创建 AudioEngine；
- 随机玩家名、助手名、家人留言；
- 初始化 Screens 和 Hud；
- 设置语言选择器与音频开关。

## 5. 游戏状态设计

### 5.1 状态集中管理

运行时主要状态由 `createInitialState()` 创建，并由 `App` 持有。

状态中包含：

- 当前屏幕：标题、首页、游戏、总结；
- 玩家位置、朝向、速度；
- 当前模式：步行 / 骑车；
- 今日状态：累 / 普通 / 精神好；
- 当前路线；
- 已投递列表；
- 当前投递动画；
- 自动导航状态；
- 输入状态；
- 漫画气泡；
- 家人留言；
- 世界随机种子；
- 世界布局和障碍；
- 高对比度 / 安心模式等辅助状态。

### 5.2 派生状态

`state/gameState.js` 提供：

- `createInitialState()`：初始状态；
- `currentTarget(state)`：当前目标住户。

原则：

- 能从已有状态计算出的值，不重复保存；
- 当前目标通过 `route` 和 `delivered` 推导；
- HUD、投递、导航都使用同一个当前目标函数，避免不同模块目标不一致。

## 6. 游戏循环

主循环在 `App.loop(now)` 中。

每帧执行顺序：

1. 计算真实时间差；
2. 如果掉帧，将一次大 dt 拆成多个小步进；
3. 更新游戏时间 `floatTime`；
4. 调用 `updatePlayer()` 更新移动、碰撞、自动导航；
5. 调用 `updateDelivery()` 更新报纸飞行动画和送达结果；
6. 更新定时陪伴提示；
7. 更新记忆提示；
8. 更新思考气泡；
9. 更新导航提示；
10. 调用 `renderer.render(state)` 绘制 3D；
11. 接收渲染层返回的环境 / 交通信息；
12. 更新音频；
13. 更新休息点提示；
14. 更新安全提示；
15. 更新 HUD；
16. 更新漫画气泡 DOM；
17. 请求下一帧。

### 6.1 分步模拟

如果浏览器卡顿，不能简单把 dt 限制到很小，否则游戏会变慢。

当前设计：

- 限制最大 elapsed；
- 将 elapsed 拆成多个 `<= 0.05s` 的小步进；
- 碰撞稳定；
- 速度不因低帧率明显缩水。

## 7. 玩法逻辑设计

玩法逻辑集中在 `src/game/`。

### 7.1 difficulty.js

负责：

- 根据步行 / 骑车生成基础答案；
- 根据今日状态生成难度配置；
- 随机选择路线；
- 在第一目标附近生成起点；
- 控制任务户数、速度、路线长度。

设计原则：

- 今日状态影响配置；
- 起点不直接贴着目标；
- 每次路线随机；
- 路线之间距离适中；
- 避免城市边缘和不可达点。

### 7.2 delivery.js

负责：

- 玩家移动；
- 步行 / 骑车速度；
- 转向；
- 碰撞检测；
- 可投递判断；
- 报纸投递请求；
- 报纸飞行动画；
- 自动导航路径；
- 交通对象碰撞；
- 安心模式影响。

核心函数包括：

- `updatePlayer(state, dt)`；
- `requestDelivery(state)`；
- `updateDelivery(state, dt)`；
- `canDeliverNow(state, target)`；
- `buildAutoNavPath(state, target)`。

## 8. 投递系统技术设计

### 8.1 投递判断

投递判断不依赖房屋碰撞中心，而使用目标的 `deliveryX / deliveryY`。

这样可以：

- 让玩家站在道路边也能投递；
- 避免必须进入院子；
- 避免围墙和建筑阻挡；
- 保持老人友好。

### 8.2 投递流程

```text
玩家按空格 / 投递按钮
  ↓
requestDelivery 检查当前目标
  ↓
若不在范围：显示温柔提示
  ↓
若在范围：创建 delivery 状态
  ↓
updateDelivery 推进报纸飞行动画
  ↓
到达目标：加入 delivered
  ↓
触发住户感谢、房屋反应、感谢卡
```

### 8.3 视觉与逻辑分离

逻辑层只负责：

- 是否可投递；
- 投递是否完成；
- 投递起点和目标点；
- 投递进度。

渲染层负责：

- 报纸模型；
- 飞行弧线；
- 旋转；
- 房屋弹跳；
- 住户头像和反馈。

## 9. 自动导航技术设计

自动导航使用道路 waypoint。

目标：

- 与地面箭头一致；
- 不穿房子；
- 不穿河；
- 不绕无意义回头路；
- 遇到交通对象可停下；
- 玩家随时可接管。

### 9.1 路径来源

路径由 `buildAutoNavPath(state, target)` 生成。

路径基于：

- 当前玩家位置；
- 道路网络；
- 当前目标投递点；
- waypoint 进度。

### 9.2 与箭头一致

渲染层的 `scenePathFromAutoNav()` 会优先使用自动导航路径。

这样：

- 自动导航实际走的路；
- 地面箭头显示的路；
- 小地图目标方向；

三者尽量保持一致。

### 9.3 避免打转

自动导航状态保存：

- `autoNavPath`；
- `autoNavTargetId`；
- `autoNavIndex`；
- `autoNavWaitUntil`；
- `autoNavBlockedUntil`。

不要每一帧重新从头规划，否则会在路口反复转向。

## 10. 世界与地图生成

地图核心在 `src/data/world.js`。

### 10.1 坐标系统

项目中有两套坐标：

- 世界逻辑坐标：用于玩家、碰撞、投递；
- Three.js 场景坐标：用于渲染。

通过 `WORLD_SCALE` 转换。

设计原因：

- 逻辑坐标可以保持较大地图范围；
- 渲染坐标适合 Three.js 尺度；
- 保留和地图数据转换的灵活性。

### 10.2 世界布局

`createWorldLayout(seed)` 根据随机种子生成：

- 建筑 lot；
- 树木；
- 地标；
- 道路；
- 水系；
- 铁路；
- 氛围信息。

### 10.3 障碍生成

`createWorldObstacles(layout)` 根据同一份 layout 生成碰撞体。

关键原则：

- 可见物和碰撞体必须来自同一布局；
- 装饰性绿化默认不加入碰撞；
- 水面阻挡玩家，桥留通道；
- 目标房屋和普通建筑都有障碍体；
- 不生成看不见的空气墙。

### 10.4 北江口数据

`kitaeguchiMap.js` 保存北江口参考地图数据，包括：

- 建筑轮廓；
- 道路段；
- 水系；
- 铁路。

游戏不会直接机械照搬真实地图，而是整理成适合老人游玩的“干净游戏版”。

## 11. 渲染系统技术设计

渲染集中在 `src/render/threeRenderer.js`。

### 11.1 ThreeRenderer 职责

- 初始化 Three.js Renderer / Scene / Camera；
- 创建地面、天空、道路、水体；
- 创建房屋、商店、地标；
- 创建玩家和自行车；
- 创建行人、动物、昆虫；
- 创建投递目标光圈；
- 创建地面导航箭头；
- 更新角色动画；
- 更新摄像机；
- 更新 LOD；
- 更新遮挡透明；
- 返回环境和交通信息给游戏逻辑。

### 11.2 场景创建流程

`createWorld()` 大致顺序：

1. 灯光；
2. 天空；
3. 地面；
4. 道路网络；
5. 目标房屋；
6. 程序化城镇；
7. 地标；
8. 环境生命；
9. 玩家；
10. 目标标记；
11. 导航箭头；
12. 报纸投射物；
13. 送达反应。

### 11.3 重建世界

语言切换或重新随机地图时，通过 `rebuildWorld()`：

- 清空场景；
- 重置缓存集合；
- 使用当前 layout 重新创建世界。

## 12. 相机设计

相机主要为第三人称跟随。

设计要求：

- 保持玩家在画面中；
- 不频繁抖动；
- 不因为骑行速度变化突然拉远；
- 避免遮挡时建筑半透明；
- 支持步行和骑车。

技术策略：

- 使用平滑插值；
- 使用玩家朝向作为相机方向参考；
- 使用 `cameraFocus`、`cameraLookTarget` 等缓存减少抖动；
- 遮挡检测使用 Raycaster。

## 13. UI 系统技术设计

### 13.1 screens.js

负责大屏页面：

- 当前区域名称。

更新频率不是每帧，而是隔几帧更新，减少开销。

## 14. 漫画气泡技术设计

漫画气泡使用 DOM 实现，而不是 3D 对象。

原因：

- 字体清晰；
- 多语言排版方便；
- 头像样式容易控制；
- 手机端半透明布局更易调整。

状态中保存：

- `comic.text`；
- `comic.tone`；
- `comic.time`；
- `comic.speaker`；
- `comic.recipient`。

`App.updateComic()` 根据状态更新 DOM。

电脑端：

- 气泡偏左；
- 半透明；
- 避开小地图。

手机端：

- 居中；
- 更小；
- 半透明；
- 不遮挡触控按钮。

## 16. 音频系统技术设计

音频集中在 `src/audio/audioEngine.js`。

设计约束：

- 浏览器需要用户手势才能播放声音；
- 标题画面点击开始时调用 `audio.start()`；
- 音乐和声效分别开关；
- 声音必须低刺激；
- 根据移动状态播放骑车 / 步行音；
- 根据环境信息播放自然声。

音频系统接收：

- 当前 state；
- dt；
- renderer 返回的环境信息。

然后决定：

- 背景音乐；
- 骑行声；
- 脚步声；
- 投递声；
- 环境声；
- 动物和交通提示音。

## 17. 多语言系统技术设计

多语言集中在 `src/i18n.js`。

### 17.1 支持语言

- `ja`；
- `en`；
- `zhHant`；
- `zhHans`。

### 17.2 语言检测

优先级：

1. URL 参数 `?lang=`；
2. LocalStorage 保存语言；
3. 浏览器语言；
4. 默认英文或中文。

### 17.3 切换语言

语言切换时：

- 更新 `locale`；
- 更新 `i18n` 字典；
- 写入 LocalStorage；
- 更新 URL 参数但不刷新；
- 更新 HTML lang 和 data-locale；
- 发送 `bicycle-language-change` 事件。

App 监听事件后：

- 更新助手名；
- 更新随机玩家名；
- 重建世界招牌；
- 刷新当前页面 / HUD / 漫画气泡。

版本号格式：

```text
YYYYMMDD.当天构建次数
```

例如：

```text
20260611.3
```

相关文件：

- `src/buildInfo.js`；
- `index.html` 中 CSS / JS cache query；
- 左上角版本号角标。

`tools/bump-version.mjs` 负责递增当天构建次数。

目的：

- 方便用户反馈截图；
- 方便确认 GitHub Pages 是否更新；
- 避免浏览器缓存旧 CSS / JS。

## 21. 性能设计

### 21.1 已使用策略

- 关闭实时阴影或降低阴影使用；
- InstancedMesh 批量渲染树、花草、重复部件；
- 建筑 LOD；
- 小地图降频更新；
- 远处低细节；
- 避免每帧重建世界；
- 控制 pixelRatio；
- 装饰物默认不参与碰撞。

### 21.2 性能风险

高风险点：

- 过多高细节建筑；
- 过多独立 Mesh；
- 行人大量骨骼式动画；
- 每帧 DOM 更新；
- 每帧重算导航；
- 每帧 Raycaster 过多；
- 手机端过高像素比。

### 21.3 后续优化方向

- 更多 InstancedMesh；
- 分区加载 / 激活；
- 更严格 LOD；
- 行人合批；
- 导航路径缓存；
- 减少透明材质重叠；
- 移动端低特效模式。

- 投递范围是否符合预期；
- 声音是否能在点击开始后播放；
- GitHub Pages 是否更新。

## 23. 部署设计

项目部署到 GitHub Pages。

流程：

1. 修改代码；
2. 运行检查；
3. bump 版本；
4. 提交 Git；
5. push 到 GitHub；
6. GitHub Pages 自动部署；
7. 请求线上页面确认版本号和缓存参数。

线上确认重点：

- HTTP 200；
- 页面版本号；
- CSS query 版本；
- JS query 版本。

## 25. 开发原则

后续改代码时遵守：

1. 不把新功能全部写进 `main.js`；
2. 玩法规则进 `game/`；
3. 数据进 `data/`；
4. DOM 页面进 `ui/`；
5. 3D 表现进 `render/`；
6. 输入只改状态，不直接操作角色；
7. 渲染层不要决定核心玩法结果；
8. 碰撞和可见地图必须同步；
9. 多语言文本不写死在逻辑中；
10. 每次部署前更新版本号并验证线上版本。
